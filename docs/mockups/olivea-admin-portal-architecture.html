System Architecture

Olivea Admin Portal

Content management, role-based access, and the road to a guest portal
April 2026 · Architecture Proposal v1

1. What we're building

A private admin portal at /admin where your team can manage every piece of content on olivea.com — text, images, menus, pricing, hours, and reservations — without touching code. Built on Supabase (auth + database + file storage) with role-based permissions, and designed so the same infrastructure powers a future guest portal.

Why Supabase? You get auth, Postgres database, file storage, and real-time subscriptions in one platform — no extra vendors. It's free-tier friendly, self-hostable if you ever need it, and the Row-Level Security (RLS) model means permissions live in the database, not scattered across app code.

The key insight: your public site stays static and fast. Content is fetched from Supabase at build time (or with ISR), so visitors still get the same blazing performance. The admin portal is a separate route group that only loads for authenticated team members.

3. Roles and permissions

Three roles, each with clear boundaries. Permissions are enforced at the database level via Supabase Row-Level Security, so they can't be bypassed even if someone finds the API.

👑
Owner
Everything — invite users, assign roles, manage all content, view analytics, configure site settings
🍳
Editor
Edit menus, pricing, images, journal posts, hours. Cannot invite users or change site settings
👁️
Viewer
Read-only access to all content. Can preview drafts. Cannot edit or publish

Future: Guest role. When we build the guest portal, we add a guest role scoped only to their reservation data, preferences, and pre-arrival forms. Same auth system, different RLS policies.

4. Database schema

Everything your site currently stores in static MDX/JSON files moves into Postgres tables. Content is bilingual (ES/EN) by default, with a clean separation between structure and translations.

Core tables

-- Users & roles (extends Supabase auth.users) CREATE TABLE profiles ( id UUID REFERENCES auth.users PRIMARY KEY, full_name TEXT, role TEXT CHECK (role IN ('owner', 'editor', 'viewer')), avatar_url TEXT, created_at TIMESTAMPTZ DEFAULT now() ); -- Page sections (hero text, about copy, etc.) CREATE TABLE page_content ( id UUID DEFAULT gen_random_uuid() PRIMARY KEY, page TEXT NOT NULL, -- 'farmtotable', 'cafe', 'casa' section TEXT NOT NULL, -- 'hero', 'intro', 'info_cards' content_es JSONB NOT NULL, -- { title, body, cta_label, ... } content_en JSONB NOT NULL, sort_order INT DEFAULT 0, status TEXT DEFAULT 'published', -- 'draft' | 'published' updated_by UUID REFERENCES profiles, updated_at TIMESTAMPTZ DEFAULT now(), UNIQUE (page, section) ); -- Menu items (wine, food, drinks, spirits) CREATE TABLE menu_items ( id UUID DEFAULT gen_random_uuid() PRIMARY KEY, menu_type TEXT NOT NULL, -- 'wine', 'food', 'drinks', 'spirits' category TEXT NOT NULL, -- 'De La Casa', 'Blancos', 'Cocktails' name_es TEXT NOT NULL, name_en TEXT, description_es TEXT, description_en TEXT, price NUMERIC(10,2), price_label TEXT, -- 'copa/glass', 'botella/bottle' winery TEXT, -- for wines grape TEXT, -- varietal info region TEXT, year INT, tags TEXT[], -- ['natural', 'house', 'seasonal'] sort_order INT DEFAULT 0, is_available BOOLEAN DEFAULT true, updated_by UUID REFERENCES profiles, updated_at TIMESTAMPTZ DEFAULT now() ); -- Media library CREATE TABLE media ( id UUID DEFAULT gen_random_uuid() PRIMARY KEY, storage_key TEXT NOT NULL, -- Supabase Storage path alt_es TEXT, alt_en TEXT, width INT, height INT, mime_type TEXT, size_bytes INT, tags TEXT[], uploaded_by UUID REFERENCES profiles, created_at TIMESTAMPTZ DEFAULT now() ); -- Banners & popups (replaces active.json files) CREATE TABLE promotions ( id UUID DEFAULT gen_random_uuid() PRIMARY KEY, kind TEXT NOT NULL, -- 'banner', 'popup' content_es JSONB NOT NULL, content_en JSONB NOT NULL, rules JSONB, -- paths, frequency, priority starts_at TIMESTAMPTZ, ends_at TIMESTAMPTZ, enabled BOOLEAN DEFAULT false, created_by UUID REFERENCES profiles, updated_at TIMESTAMPTZ DEFAULT now() ); -- Operating hours & reservations config CREATE TABLE operating_hours ( id UUID DEFAULT gen_random_uuid() PRIMARY KEY, venue TEXT NOT NULL, -- 'farmtotable', 'cafe', 'casa' day_of_week INT NOT NULL, -- 0=Sun .. 6=Sat opens_at TIME, closes_at TIME, is_closed BOOLEAN DEFAULT false, notes_es TEXT, notes_en TEXT, UNIQUE (venue, day_of_week) ); -- Journal / blog posts (replaces MDX files) CREATE TABLE journal_posts ( id UUID DEFAULT gen_random_uuid() PRIMARY KEY, slug TEXT UNIQUE NOT NULL, title_es TEXT NOT NULL, title_en TEXT NOT NULL, body_es TEXT NOT NULL, -- Markdown body_en TEXT NOT NULL, cover_image UUID REFERENCES media, author TEXT, tags TEXT[], status TEXT DEFAULT 'draft', -- 'draft' | 'published' published_at TIMESTAMPTZ, created_by UUID REFERENCES profiles, updated_at TIMESTAMPTZ DEFAULT now() ); -- Audit log — who changed what, when CREATE TABLE audit_log ( id BIGSERIAL PRIMARY KEY, user_id UUID REFERENCES profiles, action TEXT NOT NULL, -- 'create', 'update', 'delete', 'publish' table_name TEXT NOT NULL, record_id UUID, changes JSONB, -- { field: { old, new } } created_at TIMESTAMPTZ DEFAULT now() );

6. How content flows

Today (static)

MDX/JSON files → Next.js build → Static HTML → Visitor

With admin portal

Admin edits → Supabase DB → Webhook triggers → ISR revalidation → Visitor │ └─ Draft preview (admin-only, real-time)

ISR (Incremental Static Regeneration) means the public site stays statically generated and fast, but re-generates specific pages within seconds when content changes. Your visitors never see a loading spinner, and editors see changes go live almost instantly.

Migration strategy: gradual, not all-at-once

  1. Menus first. Wine list, drinks, spirits, and food menus move to Supabase. These change most often and benefit the most from a table editor. This also replaces your Canva embeds with native, SEO-friendly pages.
  2. Banners & popups. Already JSON-shaped — easy to move to the promotions table with a simple form UI.
  3. Page content. Hero text, section copy, info cards. Each section becomes an editable block.
  4. Journal posts. MDX articles move to the database with a markdown editor.
  5. Hours & reservations. Operating hours, special closures, and eventually booking config.

9. Trade-offs and decisions

Why not a headless CMS like Sanity or Contentful?

We considered it. A headless CMS gives you a polished editor out of the box. But: (a) it's another vendor and monthly cost, (b) it can't handle your menu-specific data model well (wine vintages, categories, pricing tiers), (c) it doesn't share auth with the guest portal, and (d) you lose the bilingual JSONB flexibility. Supabase gives you one platform for everything.

Why not edit MDX files directly through a Git-based CMS?

Tools like Tina CMS let non-developers edit MDX through a visual layer. But your team needs structured data (menu tables, pricing, hours) not just rich text. A database gives you proper queries, filtering, and the foundation for the guest portal.

What about the Canva menus?

The Canva embeds currently serve your wine, spirits, drinks, and pairing menus. Phase 2 replaces these with native pages rendered from the database — giving you SEO benefits, faster loads, and the ability to edit from the admin portal. During migration, both can coexist (admin-managed menus for categories that are ready, Canva fallback for the rest).

ISR vs. full SSR

ISR keeps your pages static (fast, cacheable, cheap) while allowing near-instant updates when content changes. Full SSR would hit the database on every request — unnecessary for a restaurant site where content changes a few times per week.
